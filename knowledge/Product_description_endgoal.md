* user persona: somewhat a “data person” as they need to understand the SNT stratification process, be familiar with the country data and epidemiological stuff. No need to code. * Purpose of the WebApp: provide a user friendly and intuitive interface to execute the whole SNT stratification process (= import, process, clean, manipulate data to obtain the variuos data layers needed for the next step which is the intervention mix and budgeting which happens in the SNT Explorer, which is a completely different tool)
* This SNT stratification process is broken down intp smaller steps, each step being a pipeline
* all SNT stratification pipelines already exist and work in OH. Users can create pipelines from templates (simply download to a given ws and the thing works out of the box, provided that the input data in present of course)
* Normally everything can be found in the OH ws at Pipelines section, but these pipelines are just listed in order of creation (so, not necessarily logically) and it is not very intuitive for the user to understand which ones are mandatory, facultative, alternative, … therefore, this Web App is needed provide a visually more clear overview (i.e., pipelines displayed in a vertical flow chart)
* The Web app therefore is a single (one place) interface from which the user can “see” the whole process, and execute it as well by launching the pipelines
* User should also see the status of the pipeline: was it executed ever, succeeded, fail, running, (or missing, if it is part of the SNT process but currently not installed in the ws)
	* Q: how to handle missing pipelines (not present in the ws): the flow is “standard” and universal (does not depend on ws or country), but then each ws (country) needs to take care of installing the pipelines. If they forget some, how do we display this in the Web App? One solution could be to display the pipeline in the web app, but if you try to launch it it should fail with a clear message (“Go create/install the pipeline!”). 
* The idea is that the Web App is a “recast” of the OH ws Pipelines view, but with the pipelines displayed in a nicer way: each pipeline is a “card” (name, brief description, status, last execution, label of type, and link to latest output files), and the cards are spatially arranged in the canvas as a vertical flow chart. The cards are also linked to a side view panel (right side of the screen) which opens one you click on the pipeline card and show more info (link to the pipeline’s readme.md in the GitHub repo) and allows to effectively trigger the pipeline with all the parameters. The position of each pipeline’s card is “fixed” and decided based on a map that I will develop and provide (this can then be translated to html and remain as a fixed part of the web app code)
* The user can see therefore the whole process status as a glance, + can also interact with it (launch pipelines).

* Web App UI: 
	* to be used on desktop (so, landscape view), never on mobiles
	* Centered around pipelines. Each pipeline is a “card”.
	* Pipeline “cards” display the pipeline title/name (e.g., “A.1 DHIS2 Extract”), then a very concise description (1 sentence), status (same as OH pipelines status: Succeded, Failed, Running, … also blank for not run yet) a symbol on the top left corner (green check mark, red cross, … ), type as a label on the top right corner (Mandatory, Alternative, Facultative, … ), date time of last execution (if ever executed), link to latest outputs (HTML report and if produced Dataset).
	* When clicking on the pipeline card, right-hand side view opens with the following info and functionalities: title/name, link to pipeline readme.md in GitHub repo (way more info for the user), parameters with help text (collapsible text maybe if too long), run pipeline button which triggers the pipeline (with the parameters as specified by the user), clickable link to the OH pipeline actual run (where the user can see more details like log messages)
